File Name Conventions

  • HTML files should be hyphen-delimited, e.g. antha-ui.html
  • CSS files should be hyphen-delimited, e.g. antha-ui.css (to match up with its associated HTML file)
  • ES2015 class files should be PascalCase, e.g. MyClass.js (to match the name of the class)
  • Other JavaScript files should be camelCase, e.g. someOtherCode.js
  • Markdown files should be UPPERCASE, e.g. MY_COMPONENT.md

Folder Structure

We should keep the folder structure as flat as possible - components should be siblings of one another, in general. However, it might be a good idea to adopt a component prefix convention to distinguish component behaviour, e.g.; presentational component, organisational component, data handling component, etc.

Component Conventions

  • Keep markup & CSS in one file. JavaScript should go in a separate file and imported (so it can be tested separately).
  • Use the object literal syntax for writing Polymer code, e.g. Polymer({ ... })
  • Use the underscore convention for private functions, e.g. _thisIsAPrivateFunction
  • Include a README.md with every component, giving an overview of the component, and documenting the public API

JavaScript Conventions

General

  • Strings should be enclosed by single quotes ('') only. Double quotes should only be used inside of strings
  • When testing for equality or inequality, the strict equality operators should be used: === and !==

JavaScript Conventions

Functions

  • Every function should include a comment block above its definition, describing the operation of the function, its side effects (@sideeffects), conversely whether it's pure (@pure), its parameters, return value, and info on the function author.
  • For functions with more than one parameter, each parameter should be put on its own line. (Functions with only one parameter should have their signatures all on one line).

JavaScript Conventions

Functions

  • Functions should enforce argument types and throw errors when these are incorrect. This should be done especially when an incorrect type will cause serious problems downstream.

ES2015 Classes

  • Classes should be named using PascalCase
  • Class file names should match the name of the class, e.g. MyNewClass.js
  • There should only be 1 class per file
  • Classes should contain a comment block above their definition which provides a short summary
  • static methods should be placed at the top of the class (below the constructor) in alphabetical order

ES2015 Classes

  • Private functions and variables should be prefixed with an underscore ( _ )
  • All methods should have a comment block above their definition which describes their function. This should make use of jsdoc annotations to inidicate parameters and their type, related functions, author, return value(s), access level (public, private), etc.

Laying Out HTML

  • For elements with only one attribute (or none), the opening tag and attribute should be placed on one line
  • For elements with more than one attribute, the opening tag and each attribute should be placed on their own line (this is to enhance readability, and to make changes more obvious under version control)
  • Boolean attributes (i.e. those without a value) should be grouped together at the bottom of the list of attributes
  • Function attributes (i.e. those which are passed in a function) should be grouped together
  • The class attribute should be placed immediately after the element's opening tag
